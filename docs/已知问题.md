# 已知问题

## 2026-06-26

- 当前命令行环境可以使用完整 Xcode 路径执行 `xcodebuild build`；但 `xcodebuild test` 的运行阶段会被当前沙盒限制阻止连接 `testmanagerd.control`。已改用 Xcode 的 `xctest` 直接运行构建出的测试 bundle。
- iCloud 图片是否能返回缩略图取决于系统 Photos 状态和网络可用性；如果系统无法返回图片，当前仅记录失败原因并继续加载其他图片。
- 选择“全部”时会读取图库中全部图片资产并逐张请求缩略图；大图库下可能较慢，后续阶段需要加入更完善的分页、取消和资源占用控制。
- 当前已接入 Apple Vision OCR 管线，但真实图库上的中文、英文、中英文混排、小字号和无文字图片样本尚需人工验收，不能把识别效果写成已完全验证。
- OCR 当前逐张串行处理，能避免大量图片同时解码；大图库下处理时间可能较长，后续可在确认内存表现后把并发上限提高到 2。
- 当前 OCR 会把空文本图片标记为 ready，避免无文字生活照片反复重试；后续 OCR 检索阶段需要避免把空文本当成命中。
- 当前视觉语义搜索只完成本地模型适配层、运行时 manifest 加载、模型就绪度检查、模型资源诊断 UI、图片编码器 Core ML 推理骨架、图片预处理、向量持久化、余弦检索逻辑和真实模型接入后的技术验证入口；尚未内置真实 Core ML CLIP 模型，不能完成真实图片语义搜索验收。
- 当前 MVP 路线已决定不继续把真实 Core ML CLIP 类视觉语义搜索作为第 5、6、7 里程碑的前置条件；后续产品先依赖 OCR 文本、时间和基础类型信号跑通搜索闭环。纯画面语义查询需要等合法模型、匹配 tokenizer、预处理配置和真实样本技术验证全部通过后再作为增强能力启用。
- 视觉未启用时，搜索结果不能展示伪视觉命中原因，也不能用 OCR 文本、文件名、人工标签或基础元数据冒充视觉语义搜索。
- Apple MobileCLIP 权重许可证限制为研究用途，当前不作为默认内置模型；OpenAI CLIP 是优先候选，但需要转换为 Core ML 并确认 tokenizer、预处理、体积和推理耗时。manifest 现在会把研究用途、评估用途或非商业许可证识别为不可默认启用的问题。
- `docs/视觉模型候选评估.md` 已记录 OpenAI CLIP、Apple MobileCLIP / MobileCLIP2 和 OpenCLIP 生态模型的来源、许可证风险、替代方案和影响；该文档只是接入决策依据，不包含模型文件，也不代表真实视觉语义搜索已经可用。
- `EmbeddingModelManifest.example.json` 只是本地模型接入模板，模板中的来源、许可证和输入输出名称需要在真实模型加入前逐项确认，不能把模板视为已配置模型。
- 当前 App 默认优先查找 Application Support 下的 `PictureSearch/EmbeddingModelPackage/EmbeddingModelManifest.json`，目录不存在时回退到 App bundle 中的 `EmbeddingModelManifest.json`；真实 manifest、`.mlmodelc` 文件和 tokenizer 资源仍未加入工程，因此界面仍会显示本地模型未就绪，并在模型就绪报告中单独显示 manifest 缺失或模板占位问题。
- 真实 manifest 现在必须记录外部模型引入原因、替代方案、影响评估，以及图片模型、文本模型和 tokenizer 的文件大小与 SHA-256；当前模板仍缺少这些实测值，因此会被配置校验拒绝。
- 如果真实模型资源已打包，App 会核对 `.mlmodelc` 目录和 tokenizer 文件的实际大小与 SHA-256 是否与 manifest 记录一致；大小或哈希不一致会被视为模型配置问题，需要重新确认模型文件来源、转换产物和 manifest 记录。
- App 现在会在视觉语义索引详情中展示结构化模型资源打包清单，包括图片模型、文本模型和 tokenizer 是否打包、实际大小、manifest 记录大小、SHA-256 状态和资源问题；该清单只能证明资源是否随 bundle 可见、是否与 manifest 记录一致，不能证明图文向量语义效果已经通过。
- App 现在会在视觉语义索引详情中展示可复制的 manifest 建议字段，便于把已打包资源的实际大小和 SHA-256 写回真实 `EmbeddingModelManifest.json`；这些字段只来自本地资源元数据，不代表模型来源、许可证、tokenizer 匹配性或语义效果已经通过验证。
- 当前本地模型包预检报告和 App 运行时模型就绪检查都会检查 manifest、`.mlmodelc` 目录、tokenizer 文件、大小、SHA-256，并尝试加载 Core ML 模型、检查 manifest 声明的输入输出接口、输入 shape 和输入数据类型；即使这些检查通过，也仍必须运行真实图片编码、文本编码、相似度比较和 5 张 Photos 样本技术验证，不能把预检通过写成视觉语义搜索完成。
- 目前即使用户手动加入同名 `.mlmodelc` 文件，仍需要通过 manifest 或代码配置补齐输入输出名称、tokenizer 资源名、文本序列长度、padding token ID、图片输入尺寸和归一化参数，否则模型会被视为不可用；界面现在会显示缺少的模型文件、tokenizer 和配置问题，但尚未提供自动导入或自动转换模型能力。
- 文本编码器目前已支持 token ID 输入构造、Core ML 推理骨架和本地 JSON BPE tokenizer 加载；但该 tokenizer 适配层尚未用真实 CLIP 类文本编码器验证，不能宣称已经支持 OpenAI CLIP 或 MobileCLIP 的完整 tokenizer 行为。
- tokenizer 资源会在 `EmbeddingModelManifest.json` 加载后自动解析；如果 JSON 缺失、解析失败，或 tokenizer 词表中的起止 token ID 与 manifest 记录不一致，会进入模型就绪诊断。当前仍需要真实模型提供匹配的 tokenizer 词表、merge 规则和起止 token。
- `EmbeddingValidationService` 目前只通过 mock 向量验证流程，尚未用真实 5 张 Photos 图片完成维度一致、相关查询高于无关查询、中文/英文/中英文效果对比。
- 当前已能用 `EmbeddingValidationSampleDescriptor` 和 `EmbeddingValidationSampleDescriptorDocument` 记录、解析和审计真实 Photos 验证样本，并可从 App bundle 中读取 `EmbeddingValidationSamples.json`；通过 PhotoKit 按 `assetLocalIdentifier` 加载验证图片的链路已有入口，但还没有在真实模型下跑完 5 张样本报告。审计摘要和验证报告仍只记录样本 ID，不记录 Photos 资产标识，避免把私有图库标识写入报告。
- `docs/视觉模型验证样本.example.json` 只是本机填写真实样本的模板，不包含真实 Photos 资产 ID，也不代表视觉语义模型已经通过验证。
- 当前索引状态已经能按当前模型版本显示旧向量“需重建”，但因为真实模型尚未接入，该能力还没有用真实模型升级场景人工验证。
- 当前已能生成本地图文向量技术验证 Markdown 报告，并在报告中包含模型接入清单和模型就绪诊断；但报告内容只有在接入真实模型并使用真实样本运行后才可作为验收证据，不能用 mock 报告替代真实视觉语义搜索验收。
- App 内运行技术验证后会把 `EmbeddingValidationReport.md` 保存到本机 Application Support 的 `PictureSearch` 目录；该文件只用于技术验证留档，不保存图片内容，也不应写入 Photos 私有资产标识。
- 技术验证报告现在可以记录单样本推理耗时、最小相似度差距和每张样本的实际相似度差距；但当前耗时和差距仍来自 mock 或未配置模型路径，不能代表真实 Core ML 图文模型性能。真实模型接入后必须重新记录图片编码、文本编码、总耗时、相似度差距和中英文效果。
- 当前已能生成模型与样本描述预检报告，并在 App 的视觉语义索引详情中展示技术验证预检状态；预检通过只代表可以读取样本并运行真实验证，不代表视觉语义搜索已经通过验收。
- 当前 App 已在视觉语义索引详情中提供“运行技术验证”按钮和结果摘要展示；但按钮需要模型与样本描述预检通过后才可用。未接入真实 `EmbeddingModelManifest.json`、`.mlmodelc` 和 tokenizer 时，该入口会保持不可运行或显示恢复提示，不代表视觉搜索失败。
- 当前视觉索引和视觉查询要求当前模型版本的技术验证报告或本机验证摘要已经通过；摘要会保存为 Application Support 下的 `EmbeddingValidationSummary.json`，可用于 App 重启后恢复门禁状态。摘要只保存模型版本、来源、许可证、样本数量、语言覆盖和通过状态，不保存图片内容或 Photos 私有资产标识；如果摘要损坏、模型版本不一致或真实模型文件变更但版本未更新，仍需要重新运行技术验证。
- 当前 App 已提供“视觉查询验证”入口，会用本地文本查询向量检索 SQLite 中当前模型版本的 ready 图片向量，并展示视觉相似度说明。该入口只有在真实模型就绪且已有 ready 向量后才会返回候选；未接入真实模型时不会生成伪结果。
- App 内技术验证运行时通过 PhotoKit 按样本描述读取图片，报告和界面摘要只显示样本 ID、模型诊断、分数、耗时和失败原因，不应显示 Photos 私有资产标识。
- 已新增 `docs/模型接入与验证指南.md` 作为真实模型接入和验收流程说明；该指南不是模型文件，也不代表视觉语义搜索已经通过真实验收。
- 当前视觉索引交互式批量上限为 50 张候选图片；大图库需要多次启动视觉索引才能处理完所有 pending 或需重建记录。该限制用于控制真实模型接入后的内存和响应风险，后续可在实测耗时和内存稳定后调整为可配置策略。
- 当前图片向量和文本向量推理路径已通过后台 `utility` 任务执行，并保留主线程执行诊断；但真实模型的内存峰值、首次模型加载耗时和大图库持续处理表现仍未人工验证。
- 当前搜索使用本地记录扫描和轻量字符串匹配实现 OCR 检索、时间/类型解析与融合排序，尚未使用 SQLite FTS；大图库下搜索性能和复杂分词效果需要后续真实样本评估。
- 当前本地搜索面板只展示前 5 条摘要结果，点击大图预览、结果网格和复用能力仍属于第 6 里程碑。
- 当前照片网格仅用于浏览缩略图和创建时间，点击图片尚不能打开大图预览；该能力属于后续搜索界面和图片预览里程碑。
- 当前授权成功后的 UI 已优先扩大照片网格区域，索引操作默认折叠，照片元数据改为叠放在缩略图底部，并进一步压缩侧栏宽度、网格列宽、行距和信息条高度以显示更多行；是否符合真实浏览习惯仍需在 Xcode 中用不同窗口尺寸和真实图库人工复验。
- 当前清除索引只清空本地 SQLite 记录；尚未提供索引文件大小展示、导出诊断或数据库损坏修复。
- 工程文件、Swift 类型名和系统 API 名称仍使用英文，这是代码与 Apple 工具链的正常要求；说明性文档和用户可见占位文案应使用中文。
