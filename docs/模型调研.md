# 模型调研

## 2026-06-26：本地图文向量模型候选

### 当前结论

- 当前工程尚未内置可用于产品验收的 Core ML 图文双塔模型。
- 本阶段已实现本地模型适配层、模型 manifest 解析、manifest 模板资源、Application Support 本机模型包加载、运行时 manifest 自动加载入口、模型文件大小元数据校验、模型接入清单输出、运行时 Core ML 加载和输入输出接口检查、图片编码器 Core ML 推理骨架、文本编码器 Core ML 推理骨架、图片预处理、文本 token 输入构造、向量持久化、模型版本记录和余弦相似度检索，但真实视觉语义搜索需要后续加入可分发、可运行、图文同空间的 Core ML 模型和匹配 tokenizer 后才能验收。
- 不使用人工标签、文件名、OCR 文本或手写关键词冒充视觉语义搜索。

### 候选 1：Apple MobileCLIP

- 来源：Apple `ml-mobileclip`。
- 优点：面向移动端和低延迟场景，适合本地运行；论文目标与图文检索匹配。
- 许可证影响：代码是 MIT；但模型权重使用 Apple Machine Learning Research Model License，仅限 Research Purposes，不包括商业产品开发或商业服务。因此当前不作为默认内置模型。
- 当前处理：不加入工程，不下载权重，不把能力写成已完成。

### 候选 2：OpenAI CLIP ViT-B/32

- 来源：OpenAI `CLIP`。
- 优点：图文双塔模型，图片编码器和文本编码器输出同一向量空间；许可证为 MIT。
- 集成影响：需要将图片编码器和文本编码器转换为 Core ML，打包 `.mlmodelc`，并实现 tokenizer、图片 resize/crop/normalize 等预处理；模型体积和推理耗时需要实测。
- 当前处理：保留为优先候选，但在模型文件、输入输出名称、预处理和 tokenizer 未确认前，不标记为可验收的视觉搜索能力。

### 当前工程约定

- 默认寻找 `CLIPImageEncoder.mlmodelc` 和 `CLIPTextEncoder.mlmodelc`。
- 当前模型版本为 `local-clip-unconfigured`，仅表示适配层尚未配置真实模型。
- App 启动时会优先尝试从 Application Support 下的 `PictureSearch/EmbeddingModelPackage/EmbeddingModelManifest.json` 读取本机模型包；该目录不存在时再回退到 App bundle 中的 `EmbeddingModelManifest.json`；两处都没有真实配置时降级到 `local-clip-unconfigured`，不会联网下载或假装模型可用。
- 后续真实模型可以通过本地 JSON manifest 描述模型版本、来源、许可证、模型文件名、文件大小、tokenizer 资源、输入输出名、文本序列长度和图片预处理参数。
- `Resources/EmbeddingModelManifest.example.json` 已作为模板资源加入 App bundle，用于固定真实模型接入契约；模板中的来源和许可证仍是占位说明，不能作为已验证模型使用。
- 代码会识别 manifest 模板占位内容，并把“manifest 仍是模板占位”列为配置问题；如果图片模型、文本模型或 tokenizer 的文件大小缺失，也会列为配置问题，防止误把模板或半成品 manifest 当作真实模型配置。
- manifest 可输出接入清单，记录模型版本、来源、许可证、各资源文件名、文件大小、输入输出名、图片尺寸和文本序列长度，供技术验证和人工验收留档。
- 模型缺失或配置不完整时，App 展示明确提示，不会联网下载，不会上传图片，也不会把 embedding 任务批量标记为失败。
- `EmbeddingService` 现在会生成模型就绪度报告，检查图片模型、文本模型、tokenizer 资源、输入输出名称、输入 shape、输入数据类型、文本序列长度、图片输入尺寸和归一化参数。
- 即使 `.mlmodelc` 文件存在，只要缺少输入输出和预处理配置，也不能把视觉模型标记为可用。
- 图片编码器当前支持配置驱动的 NCHW `MLMultiArray` 输入、中心裁剪、缩放、RGB 归一化和 `MLMultiArray` 输出向量解析。
- 文本编码器当前支持把 tokenizer 产出的 token ID 构造成固定长度 `MLMultiArray`，并按配置执行 Core ML 文本模型推理；工程已新增本地 JSON BPE tokenizer 适配层，可加载词表、merge 规则、起止 token 和 unknown token，但尚未用真实 CLIP 类文本编码器验证 tokenizer 行为完全匹配。
- `EmbeddingService.localPackageOrBundledOrUnconfigured` 会优先从本机模型包、再从 bundle 根据 `EmbeddingModelManifest.json` 的 `configuration.tokenizerResourceName` 自动加载 tokenizer JSON；解析失败会进入模型就绪诊断。
- 加入真实模型前必须补充：模型来源、权重许可证、文件大小、转换脚本或来源、输入输出名称、tokenizer、图片预处理规则、中英文查询效果和处理耗时。

### 技术验证入口

- 当前工程新增 `EmbeddingValidationService`，用于真实模型接入后的技术验证。
- 验证样本至少需要 5 张图片，每张图片需要一条相关查询和一条无关查询。
- 验证会检查图片向量、相关文本向量和无关文本向量维度是否一致。
- 验证会检查相关查询相似度是否稳定高于无关查询，默认相似度差值至少为 `0.05`。
- 验证样本语言需要覆盖中文、英文和中英文混合查询；缺少任一语言覆盖时，技术验证报告必须失败并列出缺失语言。
- 技术验证报告会输出模型版本、来源、许可证、样本数量、语言覆盖、缺失语言、单样本维度和相关/无关相似度，便于保留人工验收记录。
- 技术验证报告现在也会输出模型接入清单和模型就绪诊断，便于把模型文件大小、输入输出名、预处理配置、manifest 问题和缺失资源一起留档。
- 当前测试使用 mock 向量验证流程本身，不代表真实模型效果已经通过。
- `docs/模型接入与验证指南.md` 记录真实模型接入前需要确认的来源、许可证、替代方案、影响、manifest 字段和不能标记通过的情况。
