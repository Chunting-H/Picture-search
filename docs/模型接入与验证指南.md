# 模型接入与验证指南

本文用于里程碑 4「本地图文向量与视觉语义搜索」的真实模型接入和验收。当前工程不会自动下载模型，也不会把照片、OCR 文本或向量上传到网络。

## 接入前必须确认

加入任何外部模型或 tokenizer 前，先记录以下信息：

- 模型来源：权重来源、转换来源、转换脚本或转换步骤。
- 许可证：模型权重、代码、tokenizer 和转换产物的许可证；若许可证只允许研究用途，不得标记为产品可用默认模型。
- 替代方案：至少说明为什么不用系统内置能力、为什么不用更小模型、为什么不用云端模型。
- 影响：App 体积、首次加载耗时、单张推理耗时、内存占用、构建复杂度和后续维护成本。
- 隐私边界：模型必须在本机运行，不上传图片、OCR 文本、向量或索引。

## 资源放置

真实模型接入时，需要把以下资源加入 App bundle，并确认 Xcode target membership：

- `EmbeddingModelManifest.json`
- 图片编码器 `.mlmodelc`
- 文本编码器 `.mlmodelc`
- 与文本编码器匹配的 tokenizer 资源

`PictureSearch/Resources/EmbeddingModelManifest.example.json` 只是模板，不能改名后直接作为已验证模型使用。真实 manifest 必须去掉占位文字，并填写实测文件大小。

## Manifest 必填内容

真实 `EmbeddingModelManifest.json` 至少包含：

- `version`：模型版本，后续用于判断旧向量是否需要重建。
- `source`：模型来源和转换来源。
- `license`：模型权重和 tokenizer 的许可证说明。
- `imageModelName` / `textModelName`：不带 `.mlmodelc` 后缀的资源名。
- `imageModelFileSizeMB` / `textModelFileSizeMB` / `tokenizerFileSizeKB`：打包资源实测大小。
- `imageModelSHA256` / `textModelSHA256` / `tokenizerSHA256`：打包资源实测 SHA-256，用于确认模型和 tokenizer 内容没有被误替换。
- `integrationReason`：引入该外部模型或转换产物的原因。
- `alternativesConsidered`：接入前评估过的替代方案及未采用原因。
- `expectedImpact`：对 App 体积、首次加载耗时、单张推理耗时、内存占用、构建复杂度和维护成本的影响评估。
- `configuration.imageInputName` / `imageOutputName`：图片模型输入输出名。
- `configuration.textInputName` / `textOutputName`：文本模型输入输出名。
- `configuration.tokenizerResourceName`：tokenizer 资源名。
- `configuration.textSequenceLength` / `textPadTokenID` / `textStartTokenID` / `textEndTokenID`：文本 token 输入规则。
- `configuration.imageInputSize` / `imageMean` / `imageStandardDeviation`：图片预处理规则。

App 启动后会优先检查本机 Application Support 下的 `PictureSearch/EmbeddingModelPackage` 目录；该目录不存在时再回退到 App bundle 资源。两种路径都会检查 manifest、模型文件、tokenizer、文件大小、SHA-256、模型引入原因、替代方案、影响评估和文本起止 token ID。如果资源缺失、大小不一致、哈希不一致、缺少接入决策说明、tokenizer 起止 token 与 manifest 不一致或仍有模板占位，视觉模型会保持不可用状态。运行时模型就绪检查和候选模型包预检都会尝试用 Core ML 加载图片编码器和文本编码器，并检查 manifest 声明的输入输出名是否存在且为当前推理管线支持的 `MLMultiArray`。

当前推理管线要求：

- 图片编码器输入为 `float32` 的 `MLMultiArray`，shape 必须与 `[1, 3, imageInputSize, imageInputSize]` 一致。
- 文本编码器输入为 `int32` 的 `MLMultiArray`，shape 必须与 `[1, textSequenceLength]` 一致。
- 图片编码器和文本编码器输出必须是 `MLMultiArray`，输出维度是否一致会在真实样本技术验证中继续检查。

当真实模型资源已经加入 App bundle 时，可以在 App 的「视觉语义索引」详情中查看「manifest 建议字段」。这些字段会根据本机实际打包资源生成文件大小和 SHA-256，方便复制到真实 `EmbeddingModelManifest.json` 中。它们只是资源元数据辅助，不能替代模型来源、许可证、tokenizer、预处理参数和语义效果验证。

## 模型包预检

真实模型加入工程前，可以把候选资源放在同一个本机目录中，并通过 `EmbeddingModelPackageInspector` 生成预检报告。App 运行时默认读取的本机模型包目录是：

```text
~/Library/Application Support/PictureSearch/EmbeddingModelPackage/
```

目录至少应包含：

- `EmbeddingModelManifest.json`
- manifest 中声明的图片编码器 `.mlmodelc`
- manifest 中声明的文本编码器 `.mlmodelc`
- manifest 中声明的 tokenizer 资源

预检会输出：

- manifest 解析结果和配置问题。
- 每个资源是否存在、实际大小、manifest 记录大小、SHA-256 是否匹配。
- `.mlmodelc` 是否能被 Core ML 加载，以及 manifest 声明的输入输出接口、输入 shape 和输入数据类型是否能被当前推理管线使用。
- 可复制回 manifest 的大小和 SHA-256 建议字段。
- Markdown 格式的预检报告。

限制：

- 预检只证明本地资源、manifest 元数据、大小、SHA-256、Core ML 加载、基础输入输出接口、输入 shape 和输入数据类型可进入下一步验证。
- 预检不运行图片编码、文本编码或相似度比较。
- 预检通过后，仍必须使用真实 Photos 样本运行技术验证，确认图文向量维度一致，且相关查询相似度高于无关查询。

## Tokenizer JSON 格式

当前工程支持一个本地 JSON BPE tokenizer 适配层，用于把文本查询转换为固定长度 token ID 输入。该适配层不联网，不引入外部依赖，也不会使用 OCR 文本、文件名或人工标签代替文本向量。

tokenizer JSON 至少包含：

```json
{
  "vocabulary": {
    "<s>": 1,
    "</s>": 2,
    "<unk>": 3
  },
  "merges": [["h", "i"]],
  "startToken": "<s>",
  "endToken": "</s>",
  "unknownToken": "<unk>",
  "lowercasesInput": true
}
```

字段说明：

- `vocabulary`：token 到 token ID 的映射。
- `merges`：BPE 合并规则，按优先级从高到低排列，每条规则必须是两个 token。
- `startToken` / `endToken`：文本输入的起始和结束 token。
- `unknownToken`：词表缺失时的降级 token；如果不提供，遇到未知 token 会失败。
- `lowercasesInput`：是否在分词前转为小写。

限制：

- 该格式是工程内的本地 tokenizer 适配层，尚未证明与 OpenAI CLIP、MobileCLIP 或其他具体模型的官方 tokenizer 完全一致。
- 加入真实模型时，必须用该模型的 tokenizer 词表、merge 规则和预处理要求重新验证。
- tokenizer 能成功输出 token ID 只代表文本输入链路可运行，不代表图文向量语义空间已经对齐。

## 技术验证样本

真实模型加入后，至少准备 5 张 Photos 资产图片进行本机验证。样本需要覆盖：

- 中文查询
- 英文查询
- 中英文混合查询
- 生活照片或风景照片
- 截图、产品图或文档图中的至少一类

每个样本记录：

- 样本 ID
- Photos 资产标识 `assetLocalIdentifier`
- 相关查询
- 无关查询
- 查询语言
- 图片向量维度
- 文本向量维度
- 相关查询相似度
- 无关查询相似度
- 相似度差距，即相关查询相似度减去无关查询相似度
- 是否通过
- 失败原因

工程中对应的数据结构是 `EmbeddingValidationSampleDescriptor`：

```json
{
  "sampleID": "zh-1",
  "assetLocalIdentifier": "Photos 返回的 localIdentifier",
  "relatedQuery": "海边夕阳",
  "unrelatedQuery": "无关文档",
  "language": "chinese"
}
```

也可以使用 `docs/视觉模型验证样本.example.json` 作为 5 张样本描述文件模板。当前工程支持两种 JSON 形态：

- 直接保存样本数组。
- 使用 `{ "samples": [...] }` 包裹样本数组。

`EmbeddingValidationSampleDescriptorDocument` 会先校验样本数量、语言覆盖、空字段、重复样本 ID 和相关/无关查询是否相同。校验摘要只输出样本 ID 和问题原因，不输出 `assetLocalIdentifier`。

隐私注意：

- `assetLocalIdentifier` 只用于通过 PhotoKit 在本机读取验证图片。
- 技术验证报告只写入 `sampleID`，不写入 `assetLocalIdentifier`。
- 单张样本读取失败时，应记录样本 ID 和失败原因，并继续处理其他样本。

通过标准：

- 至少 5 张样本。
- 覆盖中文、英文和中英文混合查询。
- 图片向量、相关文本向量和无关文本向量维度一致。
- 相关查询相似度稳定高于无关查询，默认 margin 为 `0.05`；报告必须写出每张样本的实际相似度差距，不能只记录“通过”。
- 验证报告包含模型版本、来源、许可证、接入清单、就绪诊断、样本结果和限制。

## 不能标记通过的情况

以下任一情况出现时，不能把视觉语义搜索标记为完成：

- 只加入了 manifest 模板，没有真实 `.mlmodelc` 和 tokenizer。
- 模型文件存在，但 tokenizer、输入输出名或预处理规则未验证。
- 使用 OCR 文本、文件名、人工标签或固定关键词代替视觉 embedding。
- 只验证英文，未验证中文或中英文混合查询。
- 相关查询相似度没有稳定高于无关查询。
- 模型许可证、体积、耗时或限制没有记录。
- 只使用 mock 测试报告，没有真实 Photos 图片样本报告。

## 验收留档

真实验证通过后，需要把以下内容同步到 `docs/开发记录.md`、`docs/验收记录.md` 和 `README.md`：

- 模型版本、来源和许可证。
- 图片模型、文本模型和 tokenizer 文件大小。
- 5 张样本验证报告摘要。
- 中文、英文和中英文查询效果。
- 单张平均推理耗时和主要失败类型。
- 已知限制和降级方案。

在完成真实样本验证前，README 和验收记录只能写「模型适配层完成」或「视觉语义搜索未验证」，不得写成真实视觉搜索已完成。

技术验证通过后，App 会在本机 Application Support 的 `PictureSearch` 目录保存两类文件：

- `EmbeddingValidationReport.md`：完整 Markdown 技术验证报告，用于人工留档和问题排查。
- `EmbeddingValidationSummary.json`：最小验证摘要，用于 App 重启后恢复视觉索引和视觉查询门禁。

验证摘要不保存图片内容，不保存 Photos 私有资产标识，也不能替代完整报告和人工验收。真实模型文件、tokenizer 或预处理参数变化时，必须更新模型版本并重新运行技术验证。
