# 决策记录

## 2026-06-26: 初始工程边界

- 决策：初始工程采用原生 macOS SwiftUI App 与 XCTest 骨架。
- 原因：符合原生 macOS App 目标，便于后续接入 PhotoKit、Vision、本地存储和系统权限。
- 影响：当前只建立结构，不实现 Photos 读取、OCR、embedding 或搜索能力。

## 2026-06-26: 第二里程碑索引存储采用 SQLite

- 决策：本地索引数据结构和任务状态采用系统 SQLite 持久化，数据库位于用户 Application Support 下的 `PictureSearch/Index.sqlite`。
- 原因：SQLite 是系统可用能力，无需新增外部依赖；适合保存资产元数据、任务状态、失败原因和后续检索扩展；也便于测试去重、清除和重启恢复。
- 替代方案：JSON 文件实现更简单，但不利于大图库、并发和后续全文检索；Core Data 原生集成更强，但当前里程碑只需要明确表结构和状态流转，SQLite 更直接。
- 影响：Xcode target 需要链接 `sqlite3`；当前阶段只写入资产摘要和 OCR/向量任务状态，不保存照片原图，不启动 OCR 或 embedding。

## 2026-06-26: 第三里程碑 OCR 采用 Apple Vision 串行后台管线

- 决策：OCR 使用 Apple Vision `VNRecognizeTextRequest` 在本机执行，优先配置中文简体和英文；调度层当前逐张串行处理图片。
- 原因：Vision 是系统原生框架，不新增外部依赖，不需要网络上传；逐张处理可以降低首版内存风险，避免大量图片同时解码。
- 替代方案：第三方 OCR 或云端 OCR 可能有更高识别率或更强语言覆盖，但会增加依赖、隐私风险、许可证和网络上传问题，不符合当前里程碑。
- 影响：大图库 OCR 会较慢，但页面能显示进度、成功/失败数量、平均耗时和失败类型；后续可以在真实样本验证内存表现后把并发上限提高到 2。

## 2026-06-26: 第四里程碑暂不内置未验证图文模型

- 决策：先实现本地 Core ML CLIP 适配层、向量持久化、模型版本记录和余弦检索；在模型来源、许可证、体积、tokenizer 和预处理未确认前，不把外部模型文件加入工程。
- 原因：Apple MobileCLIP 虽适合本地低延迟场景，但模型权重许可证限制为研究用途；OpenAI CLIP 许可证较宽松，但需要转换为 Core ML 并验证输入输出、tokenizer、预处理和中文效果。
- 替代方案：使用人工标签、OCR 文本、文件名或基础元数据模拟视觉搜索会偏离 PRD，并违反“不使用人工标签代替视觉搜索”的约束；使用云端 embedding 会违反默认本机处理原则。
- 影响：当前 App 能展示视觉索引入口和模型缺失恢复提示，自动测试覆盖向量存储和相似度检索；但真实视觉语义搜索尚未验收。该决策中“不进入融合排序阶段”的限制已被 2026-06-29 的 MVP 路线调整取代：后续融合排序可先使用 OCR、时间和类型信号，视觉信号保持可选增强。

## 2026-06-28: 真实视觉模型资源必须校验大小和 SHA-256

- 决策：真实 `EmbeddingModelManifest.json` 必须记录图片模型、文本模型和 tokenizer 的文件大小与 SHA-256；App 在就绪报告中同时校验打包资源的实际大小和哈希。
- 原因：`.mlmodelc` 是目录，单看文件名或存在性无法证明模型资源与 manifest 描述一致；大小和 SHA-256 能提前发现模型转换产物替换、资源拷贝不完整或 tokenizer 指错文件的问题。
- 替代方案：只校验文件名和大小实现更简单，但无法发现同大小不同内容的资源替换；运行一次推理再判断太晚，错误提示也不够明确。
- 影响：真实模型接入前需要多记录三个 SHA-256 字段；当前没有新增外部依赖，使用系统 `CryptoKit` 在本机计算哈希。该校验只证明资源一致性，不证明图文向量语义效果通过。

## 2026-06-29: 视觉语义推理使用后台 utility 任务

- 决策：图片向量和文本向量 Core ML 推理统一通过后台 `Task.detached` 执行，任务优先级为 `utility`，并保留主线程执行诊断。
- 原因：真实图文模型加载、图片预处理和推理都可能耗时较长，不能阻塞 SwiftUI 主线程；显式执行策略可以让自动测试覆盖该约束。
- 替代方案：直接在调用方当前 actor 中运行实现更简单，但容易在 `@MainActor` 的界面状态容器中误触发主线程推理；全局并发队列也可行，但当前 Swift 并发已经满足首版逐张后台处理需求。
- 影响：视觉索引仍按交互式批量上限逐张处理，不新增外部依赖，不改变模型效果；该决策只降低界面阻塞风险，不能替代真实模型和真实 Photos 样本验证。

## 2026-06-29: 视觉模型预检校验 Core ML 输入 shape 和数据类型

- 决策：本地模型包预检在加载候选 `.mlmodelc` 后，除检查输入输出名和 `MLMultiArray` 类型外，还校验图片输入 shape 为 `[1, 3, imageInputSize, imageInputSize]` 且数据类型为 `float32`，文本输入 shape 为 `[1, textSequenceLength]` 且数据类型为 `int32`。
- 原因：当前图片预处理和文本 token 构造会生成固定形状与类型的 `MLMultiArray`。如果真实模型接口与 manifest 不一致，运行时推理会失败或产生不可解释结果，应在技术验证前尽早暴露。
- 替代方案：只在实际推理失败后提示实现更少，但错误会出现在批量索引或真实样本验证中，定位成本更高；为不同模型做自动 shape 适配会扩大当前里程碑范围，也可能掩盖预处理要求未确认的问题。
- 影响：真实模型接入需要确保 manifest 与 Core ML 模型接口严格一致；该校验不新增依赖、不运行图片编码或文本编码，也不能证明图文向量语义效果已经通过。

## 2026-06-29: 视觉模型候选优先评估 OpenAI CLIP

- 决策：里程碑 4 的真实模型接入优先评估 OpenAI CLIP 或同等 CLIP 类模型的本地 Core ML 转换产物；Apple MobileCLIP / MobileCLIP2 仅作为研究和性能参考，不作为默认内置模型；OpenCLIP 生态模型作为备选，必须逐个确认具体权重许可证。
- 原因：OpenAI CLIP 仓库许可证为 MIT，且模型结构天然符合图片编码器和文本编码器同向量空间的需求；MobileCLIP 代码虽为 MIT，但官方模型权重条款限定 Research Purposes，不适合作为默认产品模型。
- 替代方案：云端 embedding 会破坏本机处理边界；人工标签、OCR 文本、文件名或基础元数据不能替代视觉 embedding；直接引入 OpenCLIP 生态任一权重会带来许可证和 tokenizer 差异风险。
- 影响：新增 `docs/视觉模型候选评估.md` 记录来源、许可证、替代方案和影响；当前不加入模型文件、转换脚本、网络能力或外部依赖，模型版本仍为 `local-clip-unconfigured`。

## 2026-06-29: 视觉索引和视觉查询必须通过当前模型技术验证

- 决策：视觉索引和视觉查询不仅要求模型资源就绪，还要求当前模型版本的 `EmbeddingValidationReport` 已通过；未运行验证、验证失败或报告模型版本与当前模型版本不一致时，禁止生成视觉向量和执行视觉查询。
- 原因：里程碑 4 明确要求先完成本地 Core ML 图文模型技术验证，再接入完整索引流程。只检查模型文件、manifest 和 tokenizer 不能证明图文向量在同一语义空间，提前写入 ready 向量会污染后续搜索链路。
- 替代方案：允许模型预检通过后直接索引实现更顺手，但会把未验证模型的输出写入本地索引；只在文档中提醒风险不够，因为用户可能误以为视觉搜索已经可用。
- 影响：真实模型接入后需要先运行 5 张真实 Photos 样本技术验证，验证通过后才能启动视觉索引和视觉查询。当前实现会把完整 Markdown 报告和最小验证摘要保存到本机 Application Support；App 重启后可用同模型版本的通过摘要恢复门禁状态，摘要损坏或模型版本不一致时继续阻止视觉索引和查询。

## 2026-06-29: 运行时支持本机模型包目录

- 决策：`EmbeddingService` 启动时优先读取 Application Support 下的 `PictureSearch/EmbeddingModelPackage` 模型包目录；该目录不存在时再回退到 App bundle 资源。
- 原因：真实 Core ML 图文模型体积较大，接入和替换时不应强制每次都修改 Xcode 工程资源；本机模型包仍可保留 manifest、许可证、大小、SHA-256、tokenizer 和 Core ML 接口检查。
- 替代方案：只支持 App bundle 实现更简单，但会让模型迭代、许可证验证和本机试验都依赖重新打包 App；自动下载模型会引入网络和隐私边界，不符合当前约束。
- 影响：用户或开发者后续可把真实模型包放到 `~/Library/Application Support/PictureSearch/EmbeddingModelPackage/`。运行时模型就绪检查现在也会尝试加载 `.mlmodelc` 并检查输入输出接口、shape 和数据类型；但该检查仍不能替代真实 Photos 样本技术验证。

## 2026-06-29: MVP 先交付 OCR 搜索，视觉语义搜索延期为增强能力

- 决策：第 4 里程碑以“视觉能力预留、模型验证门禁和安全降级”收口；第 5、6、7 里程碑不再等待真实 Core ML CLIP 类模型，而是优先交付本地 OCR 搜索、时间/类型解析、基础融合排序、结果解释、大图预览、复用入口、手动刷新、失败重试和清除重建。
- 原因：真实视觉语义搜索需要合法模型来源、许可证确认、模型转换、tokenizer 匹配、预处理配置、Core ML 输入输出适配、5 张真实 Photos 样本验证和中英文效果验证，成本和不确定性明显高于当前 MVP 的产品收益。
- 替代方案：继续完整推进第 4 里程碑会消耗大量实现和调试成本；用人工标签、OCR 文本、文件名或基础元数据冒充视觉搜索会误导验收，也违反项目边界；使用云端 embedding 会破坏本机处理原则。
- 影响：MVP 不承诺纯画面语义查询，例如“海边夕阳”只有在真实视觉模型通过验证后才可验收。当前产品先以 OCR 文本、时间和基础类型信号形成可用闭环；视觉模型相关代码继续保留为可诊断、可验证、可安全启用的后续增强入口。
